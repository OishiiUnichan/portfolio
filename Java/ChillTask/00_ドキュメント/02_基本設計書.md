# 02_基本設計

## 2.1. システム全体構成（アーキテクチャ）

保守性と拡張性を高めるため、フロントエンドとバックエンドを明確に切り離した「APIサーバー型」のアーキテクチャを採用する。

* **フロントエンド:** HTML / CSS / JavaScript (Fetch API を使った非同期通信)
* **バックエンド:** Java (Spring Boot)
* **データベース:** リレーショナルデータベース (MySQL)

## 2.2. 画面レイアウト設計

画面遷移をさせず、1画面で完結するシングルページ構成（SPA風）の3カラムレイアウトとする。

* **【左カラム：タスク管理エリア】** タスク入力・リスト、ガチャカウンター進捗バーを表示。
* **【中央カラム：メインルームビジュアル】** アイソメトリック部屋画面、時間連動アバター、ガチャ実行ボタンを表示。
* **【右カラム：インベントリ・図鑑エリア】** 現在獲得済みの家具アセット一覧と解説テキストを表示。

## 2.3. API（通信）設計

* **`GET /api/tasks`** : タスク一覧の取得
* **`POST /api/tasks`** : 新規タスクの登録（引数: `title`）
* **`PUT /api/tasks/{id}/complete`** : タスクの完了切り替え（戻り値: ガチャ可否フラグ）
* **`GET /api/room`** : 現在の部屋の状態とアバター行動の取得（引数: 現在時刻 `hour`）
* **`POST /api/room/gacha`** : ガチャを実行し、当選家具と更新後のルームデータを返却

## 2.4. データベース（テーブル）設計

* **`tasks` テーブル（タスク管理）**  
`id` (INT, PK) / `title` (VARCHAR) / `is_completed` (TINYINT) / `created_at` (TIMESTAMP)


* **`user_room_status` テーブル（セーブデータ管理）**  
`id` (INT, PK) / `gacha_counter` (INT) / `is_demo_mode` (TINYINT)


* **`room_furnitures` テーブル（家具の配置・獲得状況）**  
`furniture_id` (VARCHAR, PK) / `area_id` (VARCHAR) / `is_core` (TINYINT) / `is_acquired` (TINYINT) / `level` (INT)
